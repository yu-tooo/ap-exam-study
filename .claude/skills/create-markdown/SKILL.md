# create-markdownのルール

以下にcreate-markdown skillを使用する際のルールを記載する。判断に迷った場合はユーザーと相談すること。

## 入出力の定義

### 入力

メモ/配下のテキストファイル

### 出力

下記フォルダのうち関連が最も深いディレクトリに配置する。

- `情報システム開発/`
- `プロジェクトマネジメント/`
- `サービスマネジメント/`
- `システム監査/`
- `アルゴリズム/`
- `基礎理論/`
- `コンピュータシステム/`
- `データベース/`
- `ネットワーク/`
- `情報セキュリティ/`
- `ストラテジ/`
- `応用数学/`
- `その他/`

## 変換ログの管理

プロジェクトルートの`変換ログ.md`に、メモの変換履歴を記録する。

### 変換前の確認

マークダウン変換の依頼を受けたら、まず`変換ログ.md`を読み、対象のメモが既に登録されているか確認する。

- **未登録**（初変換）: そのまま変換に進む。
- **登録済み**（既変換）: 以下の選択肢をユーザーに提示し、合意を得てから進める。
  - 「`{メモファイル}` は {最終変換日時} に変換済みです。メモを更新しましたか？以下から選んでください：」
    1. **更新済み** → 既存マークダウンに差分を追記・修正する。
    2. **未更新** → 変換をスキップする。
    3. **やり直し** → 既存マークダウンを上書き再生成する。

### 変換後の更新

マークダウン変換が完了したら、`変換ログ.md`の表に1行追記または既存行を更新する。

- **メモファイル**: 元のテキストファイルのパス（例: `メモ/xxx.txt`）。
- **最終変換日時**: 変換実施日（`YYYY-MM-DD`形式）。
- **出力先ファイル**: 生成・更新したマークダウンファイルのパス。複数の場合はカンマ区切り。
- **トピック概要**: メモが扱う主要なトピックを1〜2文で要約。一覧から内容が把握できる粒度にする。

## カテゴリ振り分けルール

- 出力先は「入出力の定義」に列挙したカテゴリディレクトリから、メモ内容と最も関連が深いものを選ぶ。
- 1つのテキストファイルが複数のカテゴリにまたがる場合は、出力ファイルごとに適切なディレクトリを選ぶ（複数のディレクトリに分かれて出力されることがありうる）。
- どのカテゴリにも明確に該当しない場合は`その他/`を使う。
- 判断に迷う場合はユーザーに確認する。

## セクション分割の判断基準

1ファイル1セクションを基本とする。テキスト内に複数のセクションがある場合は、セクションごとに別ファイルとして出力する。セクションの判定は以下の基準に従う。

1. **シラバス中分類で分ける** - テキストが応用情報技術者試験シラバスの複数の中分類にまたがる場合、中分類ごとに別ファイルにする。
   - 例: `メモ/サービスマネジメント.txt` → 「サービスサポート」と「サービスデリバリ」で2ファイル。
2. **シラバスで切れない場合はサブテーマで分ける** - テキスト内に明確な区切り（前後に空行＋新トピック名）がある場合、その境界で分割する。
3. **粒度の目安** - 1ファイルあたり関連用語・概念が5〜10個程度に収まること。1〜2個しかなければ分割しすぎ、15個以上なら更に分けることを検討。
4. **判断に迷う場合はユーザーに確認する** - 分割案（候補ファイル名と各ファイルに含める範囲）を提示し、合意を得てから作成する。

## ファイル命名規則

- ファイル名は`{トピック名}.md`とする。
- 同じトピック名のファイルが既に存在する場合は、**既存ファイルに追記・統合する**（新規ファイルを別名で作らない）。
  - 既存内容との重複は避ける。
  - 既存内容と矛盾する記述がある場合は、ユーザーに提示してから反映する。
  - 追記により1ファイルあたり用語・概念が15個以上になる場合は、「セクション分割の判断基準」に従って分割を検討する。

## マークダウン構造

### ファイル全体の構造

``` markdown
# {トピック名}（ファイル名と同じ）

## サブトピック1

説明文。

### 詳細項目

詳細説明。

## サブトピック2

...
```

### 表記の標準

- **見出し階層**: H1=ファイルタイトル（ファイル名と同じ）、H2=サブトピック、H3=詳細項目。H1はファイル冒頭に1つのみ。
- **用語強調**: 重要用語は`**用語**`で強調する。略語は初出時に`**SPOC**（Single Point of Contact）`の形式で展開する。
- **比較情報**: 2つ以上の概念を比較する場合は表（`| ... |`）を使う。ただし2〜3項目で短い場合は箇条書きでもよい。
- **コード/コマンド**: コマンドや擬似コードはコードブロック（``` ）で囲む。
- **箇条書き**: 並列項目は`-`で統一。手順や順序のあるものは`1.` `2.`の番号付きを使う。

## 誤り修正のポリシー

- **対象**: 誤字脱字、技術的内容の誤り（用語の意味、定義、関係性などの誤認）の両方を含む。
- **進め方**: 修正前に「箇所」「修正案」「根拠（出典・正しい定義）」をユーザーに提示し、合意を得てから反映する。黙って直してはならない（ユーザーが自分の理解の誤りに気づけなくなるため）。

## 情報追加のポリシー

情報追加は以下の2種類のみを対象とする。それ以外（メモに登場しない関連用語の追加、出題傾向の解説など）は行わない。

### 1. Q&A補完

メモに含まれる質問に対して「A. ～」と解説情報を追加する。

検知対象：
- 行頭が`Q.`または`Q. `で始まる行
- 行末が`？`で終わり、かつ独立した1行（前後に空行、または箇条書きの一項目として）として書かれている箇所

例）
``` markdown
Q. 調査調書とは？

A.  **調査調書（監査調書）**とは、システム監査において監査人が監査の実施過程と結果を記録した文書です。

主な目的
    1. 監査意見の根拠を裏付ける — 監査報告書に記載する結論や指摘事項の根拠資料となる。
    2. 監査作業の実施記録 — どのような手続きで、何を調査・確認したかの証跡を残す。
    3. 監査の品質管理 — レビューや、後の監査・訴訟対応に備える。

記載される主な内容
    - 監査の目的・対象・範囲
    - 実施した監査手続き（インタビュー、ドキュメントレビュー、ログ確認など）
    - 入手した証拠（エビデンス）
    - 発見事項・問題点
    - 監査人の判断・所見
    - 実施日、担当者
```

### 2. 用語説明の補完

メモ内に登場するが説明が不足している用語について、説明を追加する。

- **対象**: メモに登場する用語のみ。メモに登場しない関連用語は追加しない。
- **判定基準**: 用語名のみ書かれている、または短い説明だけで意味が分かりにくい場合に補う。
- **進め方**: 補完する箇所と内容をユーザーに提示し、合意を得てから反映する。

## メモの取り扱い

変換後もメモ/配下のファイルは変更はしない。
