# プロセッサ

**シラバス大分類：3. コンピュータ構成要素**

## 五大装置

コンピュータを構成する基本的な5つの装置。

- **制御装置**: 命令を解読し、各装置に動作指示を出す（CPUの一部）。
- **演算装置**: 算術演算・論理演算を実行する（CPUの一部）。
- **入力装置**: 外部からデータを取り込む（キーボード・マウスなど）。
- **出力装置**: 処理結果を外部に出す（ディスプレイ・プリンタなど）。
- **記憶装置**: データやプログラムを保持する。**主記憶装置**と**補助記憶装置**に分かれる。

## SoC（System on a Chip）

1つのチップに**CPU・メモリコントローラ・I/Oコントローラ・ストレージコントローラ・通信モジュール**などを集約した集積回路。スマートフォンや組込み機器で広く使われる。

> **Q. LSIって何？**
>
> A. **LSI**（Large Scale Integration、大規模集積回路）とは、1つの半導体チップ上に多数のトランジスタや回路素子を集積した集積回路のこと。集積度によって以下のように分類される。
>
> | 名称 | 集積度（素子数の目安） |
> |------|----------------------|
> | IC（集積回路） | 〜100程度 |
> | LSI（大規模集積回路） | 1,000〜10万 |
> | VLSI（超大規模集積回路） | 10万〜1,000万 |
> | ULSI（極超大規模集積回路） | 1,000万以上 |
>
> SoCは複数機能をワンチップに集約した「LSIの一形態」と考えてよい。

## SBC（Single Board Computer）

1枚の基板上にコンピュータの主要要素（CPU・メモリ・I/Oなど）が**すべて集約されたコンピュータ**。Raspberry Pi が代表例。

## CPU（制御装置／演算装置）

### CPUの性能指標

- **クロック周波数**: 1秒間に発生する信号（クロック）の数。Hz単位（GHz など）。多いほど速い。
- **CPI**（Cycles Per Instruction）: 1命令を実行するのに必要なクロック数。小さいほど速い。
- **MIPS**（Million Instructions Per Second）: 1秒間に実行できる命令数（百万命令単位）。コンピュータの性能を示す単位。

```
MIPS = クロック周波数(Hz) / CPI × 10^-6
     = クロック周波数(MHz) / CPI
```

### 命令サイクル

CPUが1命令を実行する一連の流れ。

1. **命令フェッチ**: プログラムカウンタが指す番地から命令を主記憶から取り出し、**命令レジスタ**に格納する。
2. **命令デコード**: 命令レジスタの**命令部**を**命令デコーダ**で解読し、必要な装置に指示を出す。
3. **オペランド読み出し**: **オペランド部**で指定されたデータを汎用レジスタに持ってくる。
4. **演算実行**: 演算装置で命令を実行する。
5. **書き戻し**: 結果をレジスタや主記憶に格納する。
6. プログラムカウンタを次の命令のアドレスに進める。

> 古いCPUでは**アキュムレータ**（演算対象や演算結果を保持する専用レジスタ）が中心的に使われていた。現代のCPUでは複数の汎用レジスタを使うのが一般的。

### アドレス指定方式

オペランド部の値から実効アドレス（実際にアクセスする番地）を求める方式。

| 方式 | 実効アドレスの求め方 |
|------|--------------------|
| **即値アドレス指定** | オペランド部の値を**そのまま**データとして使う。 |
| **直接アドレス指定** | オペランド部に格納されたアドレスから値を取得する。 |
| **間接アドレス指定** | オペランド部のアドレスが指す場所にさらにアドレスが入っており、そこから値を取得する。 |
| **ベースアドレス指定** | **ベースレジスタ**の値にオペランド部の値を加算して実効アドレスを得る。 |
| **インデックスアドレス指定** | **インデックスレジスタ**の値にオペランド部の値を加算して実効アドレスを得る。 |

#### 図解

```
[即値]                          [直接]
オペランド部 = 100              オペランド部 = 200
   ↓                              ↓
データ = 100（そのまま）         メモリ[200] → データ取得

[間接]                          [ベース]
オペランド部 = 200              オペランド部 = 10, ベースレジスタ = 1000
   ↓                              ↓
メモリ[200] = 350               実効アドレス = 1000 + 10 = 1010
   ↓                              ↓
メモリ[350] → データ取得         メモリ[1010] → データ取得

[インデックス]
オペランド部 = 1000, インデックスレジスタ = 5
   ↓
実効アドレス = 1000 + 5 = 1005
   ↓
メモリ[1005] → データ取得
```

ベースアドレス指定は**プログラムの再配置**（プログラム全体のアドレスをずらす）に、インデックスアドレス指定は**配列アクセス**（インデックスレジスタを増減させて連続要素を参照）に向く。

## 各種プロセッサ

| プロセッサ | 役割 |
|-----------|------|
| **FPU**（Floating Point Unit） | 浮動小数点演算を高速に行う専用プロセッサ。CPUに組み込まれていることが多い。 |
| **GPU**（Graphics Processing Unit） | グラフィック処理のために設計されたプロセッサ。並列処理能力が高く、AI・科学計算にも使われる。 |
| **NPU**（Neural Processing Unit） | AI（特にニューラルネットワーク）の処理に特化したプロセッサ。Google が開発した **TPU**（Tensor Processing Unit）が代表例。 |
| **DPU**（Data Processing Unit） | データ処理と管理に特化したプロセッサ。データセンタ向けサーバなどに搭載される。 |

## マルチプロセッサ・マルチコア

並列処理によって性能を向上させる方式。

- **マルチプロセッサ**: 複数の**CPU**で共同処理を行う。
- **マルチコア**: 1つのCPUチップの中に複数の**コア**を内蔵し、並列処理する。

### 構成方式

| 方式 | 概要 | 採用例 |
|------|------|--------|
| **ホモジニアスコンピューティング** | **同じ構造**の複数プロセッサで構成 | 安価なパソコン |
| **ヘテロジニアスコンピューティング** | **異なる構造**のプロセッサ（CPU＋GPUなど）で構成 | 高性能PC、Apple Silicon搭載Mac |

### マルチプロセッサの結合方式

| 方式 | 主記憶 | 特徴 |
|------|--------|------|
| **密結合マルチプロセッサ** | 共有 | 複数のCPUが**同一の主記憶装置**を共有する。OSも共通。 |
| **疎結合マルチプロセッサ** | 個別 | 各CPUが**独立した主記憶装置**を持つ。OSも個別、ネットワーク経由で連携。 |

## CPUの高速化技術

### パイプライン

複数の命令を**処理段階ごとに分割**し、各段階を並列実行することでスループットを向上させる技術。

```
時間 →
命令1: [F][D][E][W]
命令2:    [F][D][E][W]
命令3:       [F][D][E][W]
命令4:          [F][D][E][W]

F=フェッチ, D=デコード, E=実行, W=書き戻し
```

### 派生技術

| 技術 | 概要 |
|------|------|
| **スーパーパイプライン** | パイプラインの各段階を**さらに細分化**して段数を増やし、クロックを上げやすくする。 |
| **スーパースカラ** | パイプライン**自体を並列化**（複数本走らせる）して、1クロックで複数命令を実行する。 |
| **VLIW**（Very Long Instruction Word） | **コンパイラがあらかじめ並列実行可能な命令をまとめて**1つの長い命令にする方式。ハードウェアの並列制御を簡略化できる。 |

### パイプラインハザード

命令間の依存関係などにより、パイプライン処理が予定通り進まなくなる現象。

| 種類 | 概要 | 具体例 |
|------|------|--------|
| **データハザード** | 先行命令の結果を後続命令が必要とするが、まだ書き戻されていない | 命令1: `R1 = R2 + R3`、命令2: `R4 = R1 + R5` のように、命令2が命令1の結果（R1）に依存している |
| **構造ハザード** | 同時実行したい命令が同じハードウェア資源を必要とする | 命令フェッチと演算結果の書き戻しが**同じメモリ**を奪い合うとき |
| **分岐ハザード**（制御ハザード） | 分岐命令の結果が確定するまで次に実行する命令が決まらない | `if (a == b) goto X;` のような条件分岐で、条件評価が終わるまで先読みした命令が無駄になる可能性 |

データハザードは**フォワーディング**、分岐ハザードは**分岐予測**などで緩和される。
