# 分析と設計

**シラバス大分類：12. システム開発技術**

## データフローダイアグラム（DFD）

データの流れを表現する図法。以下の4要素で構成される。

| 要素 | 記号 | 意味 |
|---|---|---|
| 源泉（外部実体） | `[ ]` | システム外部のデータの発生元・送り先 |
| プロセス | `( )` | データを加工・変換する処理 |
| データストア | `┃ ┃` | データの保管場所（ファイル・DBなど） |
| データフロー | `──→` | 要素間を流れるデータ |

### 例（注文受付）

```
[顧客] ──注文情報──→ (1.0 注文受付) ──注文データ──→ ┃注文DB┃
                          │
                          └──確認メール──→ [顧客]
```

## ER図

**エンティティ**（実体）と**リレーションシップ**（関連）を表現する図法。

### 例（顧客と注文）

```
┌─顧客────────┐         ┌─注文────────┐
│ 顧客ID(PK)   │ 1    N  │ 注文ID(PK)   │
│ 氏名         │─────────│ 顧客ID(FK)   │
│ 住所         │         │ 注文日       │
└──────────────┘         └──────────────┘
```

- **エンティティ**: 管理対象（顧客・注文など）。属性を持つ。
- **PK**（Primary Key、主キー）: エンティティを一意に識別する属性。
- **FK**（Foreign Key、外部キー）: 他エンティティとの関連を表す属性。
- **多重度**（1:N など）: 関連の数の対応を表す。

## UML（Unified Modeling Language）

分析や設計で使われる14種類の記述方法をまとめた標準。代表的な図を以下に示す。

### ユースケース図

利用者の目線で、何ができるかを棒人間（アクター）を使った図で表現する。

### ステートマシン図（状態遷移図、ステートチャート図）

状態の流れを表現する。

### アクティビティ図

処理の流れを表現する。

### コンポーネント図

コンポーネント間の関係を表現する。

### クラス図

クラスの構成と他のクラスとの関係を表現する。

主な関連は以下の3種類。

| 関連 | 記号 | 意味 | 例 |
|---|---|---|---|
| **コンポジション**（合成） | ◆━━ | 全体（上位）が消えると部分（下位）も消える、強い依存関係 | 会社─部署 |
| **集約** | ◇━━ | 全体と部分の関係。上位・下位は独立できる | 図書館─本、プロジェクト─タスク・メンバー |
| **汎化と特化** | △━━ | 広い概念と狭い概念の関係（is-a関係）。子クラスは親の属性・メソッドを継承する | 動物─犬、顧客─法人顧客 |

```
コンポジション : [会社]◆━━[部署]    ※会社が消えれば部署も消える
集約           : [図書館]◇━━[本]     ※図書館が消えても本は残りうる
汎化と特化     : [動物]△━━[犬]       ※犬は動物の一種（is-a）
```

### オブジェクト図

クラスに実際に値を入れた状態（オブジェクト）を表現する。

### シーケンス図

処理を**時系列**で表現する。

### コミュニケーション図

オブジェクト間のメッセージの流れを図示する。

### 配置図

ハードウェアの配置と、その内部のコンポーネント間の関係を図示する。

### パッケージ図

関連するクラスやコンポーネント（パッケージ）をグループ化する。

## オブジェクト指向

プログラミングを小さな部品（オブジェクト）に分解する考え方。

### カプセル化

属性に直接アクセスできないように外部から隠蔽すること。

### インヘリタンス（継承）

複数のクラスに共通する属性やメソッドを親クラスにまとめ、子クラスで受け継ぐ。
