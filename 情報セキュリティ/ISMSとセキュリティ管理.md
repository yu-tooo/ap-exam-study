# ISMSとセキュリティ管理

**シラバス大分類：11. セキュリティ**

## ISMS（情報セキュリティマネジメントシステム）

組織における**情報資産**（ヒト・モノ・カネ・情報）のセキュリティを管理するための枠組み。日本語版は **JIS Q 27001**（国際規格 ISO/IEC 27001 の和訳）で規定されている。

## 情報セキュリティの3要素＋4要素

### 基本3要素（CIA）

| 要素 | 内容 |
|------|------|
| **機密性**（Confidentiality） | 認可された者だけが情報にアクセスできる状態。 |
| **完全性**（Integrity） | 情報が正確で改ざんされていない状態。 |
| **可用性**（Availability） | 必要な時に情報や機能を利用できる状態。 |

### 拡張要素（4要素）

| 要素 | 内容 |
|------|------|
| **真正性**（Authenticity） | なりすましがないこと。利用者・発信元が本物であること。 |
| **責任追及性**（Accountability） | 誰がいつ何をしたか追跡できること。 |
| **信頼性**（Reliability） | 意図した動作・結果が一貫して得られること。 |
| **否認防止**（Non-repudiation） | 行為者が後から「やっていない」と否認できないこと。 |

## リスクアセスメント

リスクについての**評価**（特定・分析・評価）を体系的に行う活動。

### 流れ

1. **リスク基準策定**: 受容する基準を決める。
2. **リスク特定**: どんなリスクがあるかを洗い出す。
3. **リスク分析**: 発生のしやすさ（発生確率）と発生時の危険性（影響度）を分析する。
4. **リスク評価**: 基準と照らして対応の優先度を決める。
5. **リスク対応**: 具体的な対応（回避・転嫁・軽減・受容）を実施する。

## 情報セキュリティポリシー

企業や組織における情報セキュリティ対策の方針や行動指針。情報資産を脅威から守るために作成する。3階層構造になっている。

| 階層 | 性質 | 内容 |
|------|------|------|
| **基本方針** | 抽象的、社外にも公開 | 組織としての情報セキュリティへの姿勢・基本理念。 |
| **対策基準** | 一般的に何をするのか | 守るべきルール・対策の指針。 |
| **実施手順** | 具体的な手順 | 個別作業のマニュアル・運用手順。 |

## デジタルフォレンジックス

**デジタル機器のデータを解析し、証拠を収集する**技術。インシデント発生後の調査や法的証拠の確保に用いる。

## 不正のトライアングル

**不正行為が発生する原因**を説明するフレームワーク。次の3要素が揃ったとき不正が起きやすい。

| 要素 | 内容 |
|------|------|
| **機会** | 不正を実行できる環境（管理の甘さ・監視の欠如など）。 |
| **動機** | 不正を働きたくなる理由（経済的困窮・恨みなど）。 |
| **正当化** | 自分の行為を正当化する理屈（みんなやっている・組織が悪いなど）。 |

## SBOM（Software Bill of Materials）

ソフトウェアに含まれる**コンポーネントの一覧表**。セキュリティ対策やライセンス管理に用いる。

- あるコンポーネントに脆弱性が見つかった場合、SBOMを参照することで**そのコンポーネントを使用している製品**を素早く特定し、対策を打てる。

## セキュリティバイデザイン／プライバシーバイデザイン

| 概念 | 内容 |
|------|------|
| **セキュリティバイデザイン** | システム・製品の**企画・設計段階から**セキュリティを確保するための方策。 |
| **プライバシーバイデザイン** | 同様に、**企画・設計段階から**プライバシー保護を確保するための方策。 |

開発の後工程で対症療法的に追加するのではなく、**最初から組み込む**ことで根本的な安全性を高める考え方。
